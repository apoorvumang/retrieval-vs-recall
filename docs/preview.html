Your search benchmark might be grading memory, not search.

A thread + the charts to go with it. Turn retrieval off on WideSearch and scores barely move — because the answers were memorized. Here's the evidence, and a fresh-data fix.

Popular "agentic search" benchmarks may be grading memory, not search.

Simplest test: how much does turning web search on actually add to a model's WideSearch score? For every serious model, a lot. For one, nothing — it goes the wrong way. 🧵

Why: WideSearch tasks are built from pre-cutoff facts — university rankings, product specs, house prices — already sitting in the model's weights. So a model can ignore the retrieved pages and just recall the answer. The score looks like search skill. It's recall.

The clean proof: run the same model closed-book (no web) vs with full search, on the public set and on fresh post-cutoff tasks.

One task makes it visceral. ws_en_034 asks for UK monthly house prices and explicitly says "cite all statistics from government websites" — a retrieval task by construction.

So we built a fresh slice from 2026 events — French Open, World Cup group stage, Cannes, Eurovision — all after the models' training cutoff. Same task format; facts nothing could have memorized.

On fresh data the illusion is gone — and the models are basically tied:

Same signature on SealQA — contamination-resistant, refreshed monthly. Answering from memory, Gemini's lead lives entirely in pre-2024 questions — the ones old enough to be in its weights. Cross into post-cutoff 2026 and the ranking flips: Mercury leads. Same context for both models — the only thing that changed is whether memory could answer.

If you benchmark search, three checks:

  1. Run a retrieval-off ablation. If the score barely moves, you're grading recall.
  2. Include post-cutoff items the model can't have memorized.
  3. Verify gold is actually retrievable, not just true.
